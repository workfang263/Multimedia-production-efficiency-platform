## 目标与验收口径（可测）

- 拼图页可选：**3 拼 / 4 拼 / 5 拼 / 6 拼**，版式固定为你提供的三张截图：
  - **3 拼**：现状（左 1 大 + 右 2 小上下）。
  - **4 拼**：左 1 大 + 右 3 小竖排。
  - **5 拼**：上排 2 大等宽 + 下排 3 小等宽。
  - **6 拼**：左 1 大 + 右 2 小竖排 + 底 3 小横排（大图始终在左侧，右侧 2 小 + 底部 3 小）。

- **画布比例**：`1:1`（现状）+ **竖版 `宽:高 = 3:4`**（宽更窄、高更长）。
- 广告页「拼图对齐模式」：增加 **`3:1 / 4:1 / 5:1 / 6:1` 下拉**，默认 `3:1`；校验规则保持你确认的语义：`商品ID有效行数 = 外链有效行数 × N`，并保持**逐组行号区间**报错（把现有 `validateStrictThree` 中的乘数与行号公式从常量 3 改为 `N`）。
- **轮播视频模式**仍按 **3 个一组**（与现网一致），**不受**拼图对齐比例下拉影响（见下方约束）。

## 总体架构思路

把「拼图」拆成两层不变量：

1. **版式层（LayoutSpec）**：每种 `pieceCount` 对应一套**格子比例与拓扑**（归一化、**不含 gutter**）；在 **`contentBox` 像素框内**划分格子并插入 **`gutterPx`（绝对 px，不随画布缩放）**，得到映射到整 `canvas` 的绝对 `cells[]`。
2. **画布外框层（CanvasFrame）**：在版式层之上套 `aspectRatio`：`1:1` 时外框正方形；`3:4` 时外框满足 `W/H=3/4`，并且版式层 **直接铺满全画布**（`contentBox = canvas`），不再做 `letterbox` 留白。

**3:4 全画布与底色（防「白边/色块」漂移）**

- `computeCells` 除返回每格 `cells[]` 外，必须返回 **`contentBox: { x, y, width, height }`**：当前定稿为 **`contentBox = canvas`**（`x=0,y=0,width=canvasW,height=canvasH`），`3:4` 下格子直接覆盖全画布，不引入上下留白带。
- 整画布底色与预览容器底色统一为 **`#FFFFFF`**（前端预览区背景、PIL `Image.new('RGB', ...)`、导出 JPEG 一致），避免出现灰底/透明底与导出图「拼在一块」的色差。

**`contentBox` 与 gutter 插入时机（定稿，防止白线随缩放漂移）**

- **不设** `contentBoxAspectRatio` 等第二份手写常数：避免与 `PUZZLE_CONFIGS` 内格子分数**手工双写**后漂移，违背「单一真相源」。
- **`gutterPx` 语义（硬约束）**：`gutterPx` = 用户配置的 **绝对像素**（默认 `4`），**不得**参与归一化坐标系的分数运算；仅在像素阶段从相邻格边界扣减。
- **`PUZZLE_CONFIGS` 归一化层**：只描述 **无 gutter** 的拓扑与各格**比例**（`[0,1]` 或列宽比/行高比 + 拓扑）；**禁止**在归一化空间「加 gutter 单位」。
- **像素阶段才插 gutter（唯一合法顺序）**：在 **`contentBox` 的像素 `width×height` 内**按拓扑做行/列划分，**相邻格之间扣除整数 `gutterPx`**；取整余数按**固定规则**吃进末行/末列或 `contentBox` 边缘，且 Vitest 断言 **相邻格净空恒为 `gutterPx`**。
- **坐标与容差（工程约束）**：优先在**相对 `contentBox` 左上角 `(0,0)`** 的坐标系内算完各格再 **`+contentBox.x / +contentBox.y`** 得到相对整 `canvas` 的绝对 `cells[]`；全画布闭合误差 ≤1px。
- **Vitest 不变量（定稿）**：对每个 `puzzleType`，在**固定 `canvas` 像素尺寸**下分别取 `aspectRatio=1:1` 与 `3:4`，断言 `contentBox` 直接等于 `canvas`，所有 `cells` 落在 `contentBox` 内，且 `3:4` 下 `cells` 覆盖画布上下边界；另抽 `p4`（或全系一种）× 两种 `canvas` 尺寸断言 **缝宽 = `gutterPx`**。

**布局计算的「单一真相源」（定稿）**

- **只在前端** [`puzzleLayout.js`](/root/work/ronghe-platform/frontend/src/utils/puzzleLayout.js)（及 Vitest）维护「格子怎么摆」的数学；请求里把结果以 **`canvas` + `cells[]` + `images[]`** 交给 Python。
- **Python 不做第二套 `computeCells`**（不维护 `puzzle_layout.py` 与前端平行实现）：`stitch_images` 只做 **白底建画布 → 校验 `cells` 合法 → 每图 RGBA 规范化叠白（§G）→ `ImageOps.fit` → `paste`**。这样改版式只改一处 JS，避免两处算歪；与「新增 3:4 画布」**有关**：3:4 只改变外框与 `contentBox` 位置，**仍由同一函数**产出 `cells`，不是两套代码各算各的。
- **大白话**：谁在算「第几张图贴左上角、宽高多少」？**只有前端那一个函数**；后端像「印刷厂」只负责按你给的裁切尺寸贴上去，并检查你没有贴出纸外。

## 数据契约（前后端必须一致）

`POST /api/video-generation/api/process/stitch` 建议扩展为（**向后兼容**：缺省新字段时服务端按 **`puzzleType=p3`、`aspectRatio=1:1`** 解析；**`gutterPx` 缺省时按 4**）。**已定稿**：**不**为「旧 p3 未改 gutter」单独走 `gutter=0` 分支——**p3～p6 与新旧会话统一默认 4px gutter**，与产品一致；历史「无缝」3 拼视觉会变化，**必须在发布说明/对内公告中写明**。

- `puzzleType`: `'p3'|'p4'|'p5'|'p6'`
- `aspectRatio`: `'1:1'|'3:4'`
- `gutterPx`: **非负整数** `number`（默认 4；UI 推荐 `2–12`，但协议层按非负整数校验）
- `canvas`: `{ width, height }` **或** `baseSize` + `aspectRatio`（二选一，落地时固定一种，推荐显式 `canvas:{w,h}` 减少歧义）
- `images`: **按固定顺序**传相对路径数组（长度必须等于 `pieceCount`），顺序与槽位定义一致（见下表）
- `cells`: **新协议必传、legacy 旧协议可缺省**。后端不维护第二套布局算法，因此 **凡是新拼图协议请求**（任一条件满足：`puzzleType in {p4,p5,p6}`、或显式携带 `aspectRatio`、或显式携带 `gutterPx`、或前端新页面提交）都必须携带 `cells:[{x,y,w,h}, ...]`，且 **`len(cells) === len(images)`**；**仅 legacy 旧三拼请求**允许缺省 `cells`，并走现有三图旧分支兼容逻辑。**禁止**出现「新协议不传 `cells`，后端再兜底算布局」。

**协议分层（定稿，给弱模型的硬边界）**

- **Legacy 旧协议**：仅服务于**现网旧三拼调用**，识别条件为「请求体不含 `puzzleType` / `aspectRatio` / `gutterPx` / `cells` 中的大部分新字段，且 `images` 仍是旧三图形态」；后端只为兼容保留现有 `p3` 旧流程。
- **V2 新协议**：凡来自本次改造后的 [`/root/work/ronghe-platform/frontend/src/views/ImageStitch.vue`](/root/work/ronghe-platform/frontend/src/views/ImageStitch.vue) 或未来扩展页面的 stitch 请求，统一视为 **V2**：**显式 `canvas + images + cells`**，并按本 plan 的单一真相源执行。
- **实现要求**：编码时先写一个 `isLegacyP3Payload(body)` 或等价判定函数，把「旧协议兼容」与「新协议严格校验」分开，避免 if/else 散落在 `stitch_images` 主流程里。

**给弱模型的最小分支规则（建议原样照做）**

1. 进入 `/api/process/stitch` 后，先 `body = request.get_json(silent=True) or {}`。
2. 先判 `isLegacyP3Payload(body)`：

   - 若是 `true`：走现有三图旧逻辑，仅补默认 `gutterPx=4` 的兼容语义，不引入 `p4/p5/p6`。
   - 若是 `false`：按 **V2 新协议** 执行，要求 `canvas/images/cells` 都齐全。

3. V2 下若缺 `canvas`、缺 `cells`、`cells.length !== images.length`、或任一 `cell` 越界，**直接 400 + `[STITCH_CELL_INVALID]`**。
4. V2 下**绝不**在 Python 内根据 `puzzleType` 重新计算布局；`puzzleType` 仅用于日志、审计、前后端联调定位。

**请求体示例（定稿，减少实现歧义）**

- **Legacy 旧三拼**：允许不带 `cells`，仅用于兼容历史调用；实现时沿用现有旧字段解析。
- **V2 新协议**：推荐固定为：
```json
{

  "aspectRatio": "3:4",
  "gutterPx": 4,

  "images": [

    "/temp/<sid>/b.png",

    { "x": 600, "y": 401, "w": 300, "h": 397 },

  ]

```


说明：示例仅表达**字段形状与职责**；具体像素值以前端 `computeCells()` 返回为准，后端不得自行推导替换。

**p6 不规则网格（定稿几何语义，避免「槽位表与文字打架」）**

- **先定底行与行 gutter（与截图一致的比例写进 `PUZZLE_CONFIGS`）**：记 **`bottomRowHeight`** 为底行在 **`contentBox` 像素阶段**分配得到的行高（整数 px）；记 **`rowGutter`** 为「上区（左大+右上两小所在区域）」与底行之间的 **行缝**，取值 **`gutterPx`**（与列缝同一绝对像素，除非产品另定）。
- **左列 1 格（下标 0）**：宽度为左列宽；**高度 = `contentBox.height - bottomRowHeight - rowGutter`**，**竖向顶对齐**到 `contentBox` 顶部（**禁止**写「占满 contentBox 全高」——底行占用 `contentBox` 底部高度后，左大不可能再占满整高，否则与「底三格横跨含左列」互斥）。
- **右上区域 2 格（下标 1、2）**：在**右列**内、**上区剩余高度**内竖排二等分（中间 gutter）；上区总高 = `contentBox.height - bottomRowHeight - rowGutter`，与左大左列同高对齐。
- **底部横条 3 格（下标 3、4、5）**：在 **`contentBox` 最下方一行**，高度 `bottomRowHeight`，横跨 **左列宽 + 列间 gutter + 右列宽**（即 `contentBox.width`），三等分（中间 gutter）。
- 右列宽 + 左列宽 + 列间 gutter = `contentBox.width`；`左大高度` + `rowGutter` + `bottomRowHeight` = `contentBox.height`。Vitest：断言格 0 底边 + `rowGutter` + 底行顶边 = `contentBox` 底部（允许 ±1px）；6 格互不重叠且均在 `contentBox` 内。

- **执行顺序（定稿，防越界与性能）**：`request.get_json()` → 解析 `canvas`/`cells`/`images` → **在调用任何 `Image.open` / 解码图片之前** 完成全部校验；校验失败则 **`print`/`logging` 一行带前缀 `[STITCH_CELL_INVALID]`**（**结构化字段**：`puzzleType`、`aspectRatio`、`gutterPx`、`canvas`、`len(cells)`、`len(images)`、`error` 短文案）并附 **`request_json_digest`**：`json.dumps(body, ensure_ascii=False)` **截断 ≤200 字符**；**`images` 路径做脱敏**（仅保留文件名或 hash 后缀，禁止整串 URL/绝对路径落盘），便于运维 `grep` 还原上下文且不泄露凭据。随后 **`return jsonify(...), 400`**。**禁止**「自动截断 cell、改坐标、继续贴图」。

## 融合平台：用户隔离现状与本迭代加固（十几人并发）

**现状（有据：代码路径）**

- **网关 `getRequestSessionId`**（[`api-gateway/server.js`](/root/work/ronghe-platform/api-gateway/server.js)）：**优先级定稿**：**HTTP Header `x-session-id` > Query `sid` > fallback `sid_shared_default`**（与现实现一致；迭代时**禁止**擅自调换顺序）。**拼图下载** `/api/fetch-image` 将文件写入 `public/temp/{sessionId}/`，静态挂载为 `/temp/...`，故 **sessionId 相同的客户端会共享同一目录**。
- **网关内存结构**：如 `imageLinkStore`、`batchDataStore` 等以 `sessionId` 为 key 分桶（同文件靠前段落），避免不同会话串数据。
- **video-service `get_user_session_id`**（[`video-service/app.py`](/root/work/ronghe-platform/video-service/app.py)）：优先 **Header `x-session-id`**，否则 Cookie/Flask `session`，用于上传目录与 stitch 读盘路径解析；网关代理 `/api/video-generation` 时会把 **`x-session-id` 从上游请求透传**（`api-gateway/server.js` 代理段）。
- **前端 Pinia 持久化**：广告/拼图表单状态存 **浏览器本地**，天然按终端用户分离，**不**经服务器多租户合并。

**本迭代必须加固（与拼图直接相关）**

- **拼图页 axios 须带 `x-session-id`，且与工具页同源（定稿）**：**同一浏览器 profile 内**，`ImageStitch`、`VideoGeneration`、广告批量链里凡命中 `/api/fetch-image`、`/api/video-generation/.../stitch`、读 `/temp/...` 的请求，**必须使用同一个 `getSessionId()` 实现**（读写的 `localStorage` key **完全一致**；首次访问无 key 时创建一次 UUID 并 `setItem`，之后只读）。否则拼图页写入 `public/temp/{sidA}/`，表格/视频页带 `sidB` 去读，**路径永不对齐**，外链回写也无法与生成侧共用临时文件。多人场景下仍靠 **不同浏览器/隐身 = 不同 localStorage → 不同 sid** 隔离；**禁止**各页面各自 `Math.random`/`Date` 造 session。
- **进程内单例缓存（定稿，防同一 JS 运行时重复读写过载/竞态）**：[`utils/session.js`](/root/work/ronghe-platform/frontend/src/utils/session.js) 中 **`let cachedSid = null`**，`getSessionId()` **先返回 `cachedSid`**，否则 `localStorage.getItem(SESSION_KEY) || uuid()`、`setItem`、`cachedSid = sid`。效果：同标签页、同模块多次初始化时 **sid 恒定**；与 `storage` 事件跨标签同步属于可选增强，本迭代**至少**满足单例 + localStorage。
- **工程验收（定稿）**：本地 **窗口 A（正常）+ 窗口 B（隐身）** 各生成一次 `fetch-image` 与 `stitch`，核对宿主机或容器内 `public/temp/` 下 **仅各自 `sessionId` 子目录** 出现新文件；**B 的请求不得读取 A 目录下路径**（请求体 `images[]` 必须落在本会话子路径）。通过后再认为「十人并发」在拼图链路上达标。
- **并发**：单进程 Node + Flask 以 **异步 I/O + 每请求独立文件路径** 为主；十人量级通常足够；若 stitch 同 session 同毫秒撞名，沿用现有时间戳文件名策略；**禁止**跨用户读写 `/temp` 下非本 `sessionId` 路径（后端路径解析已按 session 子目录，前端不得传他人路径）。

**与「3:4 画布」关系**：隔离是 **「谁的数据写哪个文件夹」**；3:4 是 **「画布外框比例」**。两者独立，但 **session 一致** 才能保证「我下载的图」和「我 stitch 读的图」在同一目录下被同一 worker 打开。
