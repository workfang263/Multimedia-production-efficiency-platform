# 云端项目总结（2026-04-02）

> 说明：本文件用于收口本轮云端开发窗口，统一记录 M2（FB 钉钉登录闭环）与 M2.5（融合平台图床容器化）阶段的目标、过程、问题、解决方法、当前进度与下一步边界。  
> 唯一执行基线仍然是：`/root/work/.cursor/plans/终版云本地协同方案_6100b6d5.plan.md`

## 一、这轮云端项目到底在做什么

这轮不是“零散修 bug”，而是在完成两条云端主线：

1. `FB-Ad-Logic-Engine` 的 M2：把“钉钉接口能调通”升级成“真正可登录、可绑定、可审计、可维持会话”的产品闭环。
2. `ronghe-platform` 的 M2.5：把融合平台图片外链上传，从历史旧图床链路切到 `tuchuang-backend` 内网服务，并让主域名下访问、上传、预览都稳定可用。

从架构视角看：

- M2 解决的是“你是谁，能不能进 FB”。
- M2.5 解决的是“融合平台生成出来的图片，能不能稳定上传并回显给前端使用”。

前者是认证闭环，后者是资源链路闭环。两者都属于“云端基础能力”，所以优先级高于本地业务页面开发。

## 二、项目总目标没有变

项目总目标始终是这三步：

1. 先在云端把基础设施、入口、认证链路、资源链路打稳。
2. 再在本地分支完成业务功能开发和交互优化。
3. 最后统一在云端收口，一次性上线钉钉扫码、融合平台能力和本地新功能。

这也是为什么本轮一直强调“总闸”“灰度开关”“不要提前全量放开”。

## 四、本窗口遇到的主要问题与解决方法

下面按“问题 -> 原因 -> 解决方案”的方式总结。

### 问题 1：钉钉登录接口能调，但系统并没有真正登录成功

原因：

- 只有“拿到钉钉用户信息”还不够，系统还需要把外部身份映射到本地用户、建立会话、返回给前端可识别的登录态。

解决：

- 补齐回调处理、身份映射、待绑定态、绑定/注册分流、JWT/cookie、`/api/me` 整体闭环。

### 问题 2：`DINGTALK_STATE_MISMATCH`

原因：

- 浏览器在 HTTP 环境下不会按预期带回 `Secure` Cookie，导致回调校验时读不到原始 `state`。

解决：

- 在当前 HTTP 调试阶段明确允许 `AUTH_COOKIE_SECURE=false`。
- 保留将来切 HTTPS 后再切回更安全配置的路径。

### 问题 3：进入 `/bind` 后又跳回 `/login`

原因：

- 后端逻辑已经更新，但前端 `dist` 还是旧包，浏览器实际拿到的还是旧页面逻辑。

解决：

- 重新执行前端构建并重启 `fb-ad-nginx`，让最新静态资源生效。

### 问题 4：门禁模式没有“直接跳钉钉”，而是先看到登录页

原因：

- 初版实现只是“显示门禁提示”，并没有在前端挂载时立即跳转到钉钉入口。

解决：

- 修改登录页和注册页逻辑：当 `ENABLE_DINGTALK_AUTH=true` 且 `ENABLE_DINGTALK_GATE=true` 时，页面加载后直接跳转到钉钉授权页。

### 问题 5：融合平台图片上传返回 `404 Cannot POST /api/upload-image`

原因：

- 融合平台前端虽然已经改成调用 `/api/upload-image`，但主域名入口层 `fb-ad-nginx` 还没有把这个路径转发给 `ronghe-nginx`。

解决：

- 修改 `FB-Ad-Logic-Engine/deploy/nginx-fb-ad-docker.conf`。
- 将 `upload-image` 与相关静态资源路径纳入融合平台分流规则。
- 重启 `fb-ad-nginx` 使配置生效。

### 问题 6：浏览器里导入的图片不显示、拼图结果图片也不显示

原因：

- 浏览器请求的是 `/temp/...`、`/uploads/...` 根路径，但主域名入口没有继续把这些静态文件路径正确代理到融合平台。

解决：

- 在主域名入口层补上 `/temp/` 和 `/uploads/` 到 `ronghe-nginx` 的反向代理。

### 问题 7：后端 `curl` 验证已成功，但浏览器页面还在报旧错误

原因：

- 浏览器缓存了旧的前端构建产物，仍在执行旧 bundle。

解决：

- 强刷页面或清理浏览器缓存后重新加载。
- 这类问题的本质是“浏览器代码版本”和“服务器代码版本”不一致。

### 问题 8：M2.5 过程中旧图床链路和新图床链路同时存在，容易混淆

原因：

- 历史上保留过 `imgfi` 相关接口、注释、文案和 `oss-upload-service` 编排。
- 迁移到 `tuchuang-backend` 后，如果旧链路不清理，后续排错会不断误判。

解决：

- 删除旧 `/api/upload-to-imgfi` 活接口。
- 移除 `docker-compose.yml` 中旧 `oss-upload-service` 编排入口。
- 把活代码中的日志、文案、注释统一切换为“统一图床 / 外链”口径。

### 问题 9：容器重建后出现 `502 Bad Gateway`

原因：

- `api-gateway` 容器在重建后更换了新的容器 IP。
- `ronghe-nginx` 仍然持有旧的上游地址缓存，导致请求被转发到失效地址。

解决：

- 重启 `ronghe-nginx`，让它重新解析 `api-gateway` 服务名。
- 重新验证主域名下的 `/api/upload-image`，确认返回恢复为网关自己的 JSON 响应，而不是入口层 `502`。

## 五、现在云端架构的正确口径

### 1. FB 认证口径

- 采用 `FB 后端直连钉钉 OAuth2`。
- 不再把 `oauth2-proxy` 作为当前主线方案。
- FB 自己维护：
  - 回调
  - 身份映射
  - 登录审计
  - JWT/cookie 会话
  - `/api/me`

一句话理解：

钉钉负责回答“这个人是不是公司里那个员工”，FB 负责回答“这个人映射到我系统里的哪个账号，以及他在我系统里能做什么”。

### 2. 融合平台图床口径

- 前端只认 `api-gateway`。
- 网关只暴露统一入口 `/api/upload-image`。
- 实际上传由内网服务 `tuchuang-backend` 完成。
- 图床容器不直接暴露公网端口。

一句话理解：

前端不需要知道图床是谁，只需要把文件交给网关；网关再把文件转交给真正的图床服务。

### 3. `app-network` 的正确理解

`app-network` 是“云端服务互通骨架”，不是认证中心，也不是用户系统。  
它的作用是让 `fb-ad-nginx`、`ronghe-nginx`、`api-gateway`、后续认证探针等容器之间可以用服务名互相访问。
